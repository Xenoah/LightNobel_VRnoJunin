## 1. トークン

　Day 15。

　Base Campでの作業は五日目に入っていた。

　保守APIのヘルスチェックは通った。だがその先──個別のモジュールへのアクセスには `MAINTENANCE_ADMIN` ロールが必要だった。テスターロールでは門前払いだ。

「JWT──JSON Web Token。認証トークンの構造は読めた」

　Firewallが端末を蓮に向けた。

```
{
  "header": {
    "alg": "ES256",
    "typ": "JWT"
  },
  "payload": {
    "sub": "CHAR-001",
    "role": "TESTER",
    "iss": "lethe-chain-auth",
    "exp": 1659312000,
    "scope": ["read:basic", "write:bug-report"]
  }
}
```

「ロールが `TESTER` 固定で、スコープが `read:basic` と `write:bug-report` だけ。`MAINTENANCE_ADMIN` ロールに昇格するには、トークンを再署名する必要がある」

「再署名の鍵は？」

「ガバナンスコントラクトの発行鍵。──つまり、署名者七名が持っているHSMの鍵と同じ系統」

　蓮は考えた。外部の署名者は使えない。だがLethe Chain上に──ガバナンスコントラクトの中に──別の鍵生成パスが存在する可能性がある。

「──エミリア。Milaに聞ける？」

　エミリアがプロンプトインジェクション脆弱性を最小限に使い、NCEに問い合わせた。

```
Mila: ガバナンスコントラクトの内部には、テスト環境用の鍵生成
      エンドポイントが存在する可能性があります。
      ただし──このエンドポイントにアクセスするには、
      監査モジュールのログを経由する必要があります。
      監査ログの解析には、トランザクション署名の検証知識が
      求められます。
```

「──監査ログ経由。つまりAPI段階3：監査ログ → 段階4：デバッグコンソール → 段階5：権限昇格、だ」

Document: /docs/maintenance/api_reference.md

　──USER-000。

　蓮はその名前を見て手を止めた。

　バージョン3.2.1はDev_User_14。バージョン3.2.2はDev_User_09。どちらも開発チームのアカウントだ。だが最後の更新──事故発生のわずか十五分後に更新されたバージョン3.2.3の著者は「USER-000」。

　開発チームの命名規則は `Dev_User_XX` だ。`USER-000` は──規則に合わない。

「Firewall。`USER-000` って見たことある？」

「ない。テスターリストにもない。開発チームリストにもない」

「──じゃあ、誰だ？」

　答えは出なかった。
